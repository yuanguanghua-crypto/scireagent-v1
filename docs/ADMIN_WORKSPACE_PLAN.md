# SciReagent 管理后台模块规划 — 研究员工作台

## 设计原则

> **研究员进管理后台只办三件事：上新产品、关联知识、更新数据。**
> 所有操作在 1-2 个页面内完成，不允许跨 3 个以上页面跳转。
>
> 研究员是老手，全字段表单比分步引导更高效。
> 分子结构图渲染在 admin 内实时预览，后端 RDKit + 前端 JS 双方案。
> 知识内容来自外部 Agent 输出的 JSON，admin 只做导入和确认，不做 AI 生成。

## 一、导航分组

从现在的 6 组缩到 **3 组**：

| 分组 | 研究员可见 | 说明 |
|------|-----------|------|
| **🟦 产品中心** | ✅ | 研究员主要工作区 |
| **🟩 知识中心** | ✅ | 研究员管理知识内容 |
| **⬜ 系统管理** | ❌ 仅管理员 | 订单、用户、Token 等（现有 Commerce/Accounts/System 合并） |

### 现在的 6 组 → 3 组

```
现状:                             建议:
Products                           🟦 产品中心
Knowledge                             产品（含 SKU inline + 关联方法 inline）
Relationships                         产品导入
Commerce                              分类管理
Accounts                          
System                           🟩 知识中心
                                   研究目标、应用、方法、协议、文献
                                   知识链快速创建（新增）

                                  ⬜ 系统管理
                                   订单、用户、组织、文件、Token 等
```

## 三、产品导入（批量新增）

这是研究员另一个常用功能。当前没有批量导入界面。

```
┌─────────────────────────────────────────────────┐
│  产品批量导入                                    │
├─────────────────────────────────────────────────┤
│                                                 │
│  步骤 1：下载模板                                │
│  [Download CSV Template] — 包含所有字段的模板     │
│                                                 │
│  步骤 2：上传填写好的 CSV                        │
│  [Upload] — 自动校验格式和必填项                  │
│                                                 │
│  步骤 3：预览匹配结果                            │
│  │ Row │ Name │ CAS │ 系统匹配的知识 │ 状态    │ │
│  ├─────┼──────┼─────┼───────────────┼────────┤ │
│  │ 1   │ ATP  │ 56-…│ ✓ 已匹配方法   │ 待导入 │ │
│  │ 2   │ GTP  │ 56-…│ ⚠ 未匹配知识   │ 待确认 │ │
│                                                 │
│  [Import All] [Import Selected]                 │
└─────────────────────────────────────────────────┘
```

这个导入功能的核心是**知识自动匹配**——上传的 CSV 中的产品名称/CAS 号会与现有知识实体（Methods、Applications）做模糊匹配，预先建立关联建议，研究员只需要确认或修正。

## 四、知识导入（Agent 输出 JSON 导入）

这是从 BioProAgent 输出的完整知识图谱 JSON 的导入界面。JSON 结构如下：

```
知识图谱 JSON (knowledge_graph_v3.json)
├── ResearchGoal (8)  → 引用 applications
├── Application (8)   → 引用 goals
├── Method (10)
├── Protocol (88)     → 引用 products
├── Product (110)     → 引用 protocols, applications, 含 skus
└── SKU (330)         → 引用 products
```

每个实体通过唯一 ID 字符串互相关联（如 `"product_sc8001"`、`"goal_001"`、`"protocol_0050"`）。

### 导入流程

```
┌─────────────────────────────────────────────────┐
│  知识导入（从 Agent JSON）                        │
├─────────────────────────────────────────────────┤
│                                                 │
│  步骤 1：上传文件                                │
│  [Upload JSON] — 必须是 knowledge_graph_v3 格式  │
│                                                 │
│  步骤 2：解析预览                                │
│  ┌────── 导入预览 ──────────────────────┐       │
│  │  ✅ ResearchGoal: 8 条（2 条新增）    │       │
│  │  ✅ Application:  8 条（全部已存在）   │       │
│  │  ✅ Method:       10 条（3 条更新）   │       │
│  │  ⚠️ Protocol:     88 条（80 条新增） │       │
│  │  ✅ Product:      110 条（5 条更新）  │       │
│  │  ✅ SKU:          330 条（10 条新增） │       │
│  └──────────────────────────────────────┘       │
│                                                 │
│  步骤 3：逐项确认变更（可批量确认）               │
│  │ 实体 │ 类型 │ 变更 │ 详情                  │ │
│  ├──────┼──────┼──────┼────────────────────┤ │
│  │ ATP  │ Prod │ 更新 │ 新增 2 个协议关联    │ │
│  │ GTP  │ Prod │ 新增 │ 新的产品 + 3 SKU    │ │
│                                                 │
│  [Import Selected] [Import All]                  │
└─────────────────────────────────────────────────┘
```

### 导入顺序（系统自动处理实体依赖）

```
步骤 A: ResearchGoal → 先创建/更新（不依赖其他实体）
步骤 B: Application → 关联到 ResearchGoal
步骤 C: Method
步骤 D: Protocol → 关联到 Product
步骤 E: Product → 关联到 Protocol、Application
步骤 F: SKU → 关联到 Product
```

### 与产品导入的区分

| 功能 | 产品导入（CSV） | 知识导入（JSON） |
|------|----------------|-----------------|
| **输入** | 研究员手填或整理的 CSV | Agent 自动生成的 JSON |
| **内容** | 产品基础信息 + SKU | 完整的知识图谱（Goal → Product） |
| **匹配策略** | CAS/名称模糊匹配 | ID 精确对应 |
| **使用场景** | 日常少量新增 | Agent 跑完一批后批量导入 |

### 与产品编辑页的关系

产品导入和知识导入都会创建/更新 Product 记录。两者的关系是：

- **Agent JSON 导入** → 创建产品 + 知识关联（一次完成）
- **CSV 导入** → 只创建产品基础信息，知识关联留空或后续补充
- **手动编辑** → 补充或修正以上两种方式的结果

建议优先级：研究员的日常工作流是 **Agent 跑一批 → JSON 导入 → 手动微调**。CSV 导入只作为 Agent 不可用时的降级方案。

### 现有实体管理（保留当前配置）

- 研究目标

- 科研方法

## 七、实施路线

| 阶段 | 内容 | 工作量 | 优先级 |
|------|------|-------|--------|
| **1** | 给 ProductAdmin 加 ProductMethodInline、ProductReferenceInline、兼容产品 Inline | 1 天 | **P0** 研究员买完即刻见效 |
| **2** | 隐藏研究员不需要的导航分组和 SKUAdmin 独立入口 | 0.5 天 | **P0** 简化界面，跟随阶段 1 |
| **3** | SMILES → 结构图实时预览（admin widget，RDKit + JS 双方案） | 2 天 | **P1** 研究员新建产品时的核心体验 |
| **4** | 知识导入（Agent JSON 上传 → 解析 → 预览 → 写入） | 3-4 天 | **P1** 承接 Agent 输出，跑通完整链路 |
| **5** | 研究员 Dashboard（待办 + 最近更新 + 快捷操作） | 1-2 天 | **P2** 研究员日常使用的门户 |
| **6** | 知识链快速创建（一次性填完所有层级） | 2-3 天 | **P2** 补缺场景，非核心路径 |
| **7** | CSV 批量导入产品 | 2-3 天 | **P3** Agent 不可用时的降级方案 |

> **阶段 1-2 是"零成本提效"**——只改 admin.py 几十行代码，研究员立刻不用再跳页面。
> **阶段 3-4 跑通核心链路**——Agent → JSON → admin → 数据库，形成完整闭环。
> **阶段 5-7 锦上添花**——等工作流稳定后再优化体验。
